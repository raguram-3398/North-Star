Companion to PRD_North_Star.md. This document defines *how* the system is built: stack, data model, component boundaries, and explicit out-of-scope decisions ‚Äî so implementation choices are made once, here, not improvised turn-by-turn during the build.

## 1. Stack

| Layer | Choice | Notes |
|---|---|---|
| Language / agent framework | Python + Google ADK | Per course, per existing setup |
| UI | **Streamlit** | Native multi-step session-state model fits the intake‚Üíoutline-confirm‚Üíday-by-day flow better than a single input/output demo tool; deploys natively on HF Spaces |

| LLM | Gemini (via ADK) | Per course |
| Deployment | HF Spaces (Streamlit SDK) | Public link satisfies project-link requirement with no login |
| Secrets | HF Space secrets | Neon connection string, Tavily key, Gemini key ‚Äî never committed to repo |

**Resolved implementation details (flagged during `src/db/connection.py` / `src/data/roles_cache.py` implementation, not pinned down above at the time of writing):**
- **DB access library**: plain SQLAlchemy 2.0 (declarative `Mapped`/`mapped_column` style), not SQLModel ‚Äî the table above left this open ("SQLAlchemy (or SQLModel)").
- **Driver**: `psycopg` v3 (`postgresql+psycopg://` SQLAlchemy dialect), not the legacy `psycopg2` a bare `postgresql://` connection string resolves to by default ‚Äî `pyproject.toml` installs `psycopg[binary]>=3.2.0`, not `psycopg2`. `db/connection.py` rewrites a bare `postgresql://` prefix automatically so `.env`'s existing `NEON_CONNECTION_STRING` format doesn't need to change.
- **Neon timeouts** (CLAUDE.md guardrail #14 requires an explicit timeout on every external call, but names no duration): `connect_timeout=10` seconds (TCP handshake) and `statement_timeout=10` seconds (per-query, set via a Postgres connection option) ‚Äî both generous-but-bounded defaults for a free-tier instance, to be tuned once real latency is measured (ship-day README requirement).
- **Engine lifecycle**: the Engine is a lazy-but-memoized module-level singleton in `db/connection.py` (created on first use, cached, never recreated) rather than instantiated at raw import time ‚Äî this avoids making `NEON_CONNECTION_STRING` a hard import-time requirement (e.g. in CI/test contexts with no DB configured) while still satisfying "one client per module."
- **`mcp` (the official Python MCP SDK) is a required explicit dependency, not something `google-adk` pulls in on its own.** `google.adk.tools.mcp_tool.mcp_toolset` imports `mcp` directly (`ClientSession`, `streamablehttp_client`, etc.), but `google-adk`'s own package metadata does not declare it ‚Äî importing ADK's MCP toolset fails with `ModuleNotFoundError: No module named 'mcp'` until `mcp` is installed and added to `pyproject.toml` separately. Discovered during the Himalayas MCP connectivity spike (`tests/spike_grounding_connectivity.py`); resolved by adding `mcp>=1.28.1` to `pyproject.toml`.
- **Himalayas MCP's tool responses are not structured JSON, contrary to this table's original "structured" description.** Every tool call (`search_jobs`, `get_salary_data`, etc.) returns `{"content": [{"type": "text", "text": "<markdown/emoji-formatted prose>"}], "isError": bool}` ‚Äî a human/LLM-readable text blob, not named fields. `search_jobs`'s text is a series of `üöÄ`-prefixed listing blocks, each with a title, company, an optional "üõÝÔ∏è Key Skills:" line (bullet-separated, `+N more` truncation suffix on the last item), and an "Apply on Himalayas" URL. `src/data/himalayas_parser.py` is the deterministic pattern-extraction module that turns this text into structured `ParsedJobListing` objects (`title`, `company`, `skills`, `source_url`) for Agent 1's cross-validation to consume ‚Äî see that module's docstring and `tests/fixtures/himalayas_search_jobs_*.txt` (real captured samples across 4 seed roles) for the full shape. `get_salary_data`'s text has a different, aggregate-statistics shape with no skills and no per-entry URL (only one shared URL for the whole response) ‚Äî parsing that is out of scope for `himalayas_parser.py`, which targets `search_jobs` only.
- **`src/data/tavily_parser.py` extracts skills from Tavily's `content` field** ‚Äî structurally the inverse problem from Himalayas: Tavily's response envelope is already per-result JSON (`url`/`title`/`content`/`score`, no blob-splitting needed, `url` used directly as `source_url`), but `content` itself is unstructured free-text prose (career-advice articles, job-description templates, forum posts, even a YouTube transcript ‚Äî see `tests/fixtures/tavily_search_*.json`, real captures for the same 4 seed roles as the Himalayas fixtures) with no equivalent of Himalayas's consistent "Key Skills:" line. `parse_tavily_result`/`parse_tavily_response` extract skills via case-insensitive, word-bounded matching against a fixed, evidence-based vocabulary (`TECH_SKILL_VOCABULARY`, ~94 terms derived from skills `himalayas_parser.py` actually extracted for these same 4 roles, plus 2 terms observed directly in the Tavily content). This is explicitly a coarser, less trustworthy mechanism than `himalayas_parser.py`'s structural parsing ‚Äî flagged in the module's own docstring, not presented as equally reliable. **Real-data finding:** Tavily's own `score` does not reliably predict whether a result's `content` has anything extractable at all ‚Äî the single highest-scoring result across all 4 fixtures (Indeed's Data Analyst page, score 0.87) is a sitemap-style list of unrelated job titles with zero extractable skills, while several lower-scoring results on the same query name concrete tools explicitly. `score` is passed through on `ParsedSearchResult` untouched (no filtering inside this module) ‚Äî what to do with that finding (e.g. whether/how `data/cross_validation.py` should weight or threshold on it) is left to a follow-up task, not decided here.

## 2. Orchestration Principles (course-aligned)

Per course guidance on DAG orchestration and "Shift Intelligence Left": agents hold reasoning and generation only. Deterministic computation is pulled into plain, testable modules that agents call as tools ‚Äî never left as an instruction the agent is expected to remember and apply correctly at runtime. Concretely, the following are **not** agent logic, regardless of how they were described at the requirements stage ‚Äî they are plain functions:

- Confidence-ladder tier assignment and source/schema validation (`security/output_guard.py`)
- Clarify-gate bound-counting, loop-termination, and first-pass real/vague/nonsense stated-goal classification (`security/input_gate.py`)
- Outline-confirmation bound-counting and loop-termination (`security/input_gate.py`'s `OutlineConfirmationState` ‚Äî a second, independent bounded loop living alongside the clarify gate's, per CLAUDE.md guardrail #8's explicit naming of both)
- Significant-event detection ‚Äî bucket/confidence-crossing diff (`outline/significant_event.py`)
- Pace calculation ‚Äî topic_score, timing_ratio, the 80/20 blend, sustained-drift threshold check (`pace/calculator.py`)
- Patch-note confidence branching (prioritize vs. ask-user, a threshold lookup on an already-computed value) (`patches/patch_manager.py`) ‚Äî implemented as `branch_by_confidence` returning the literal `"prioritize"` / `"needs_user_decision"` (only `high` confidence prioritizes; see PRD ¬ß7.9 for the resolved cutoff)
- Outline insertion/positioning into an *already-known* dependency structure (`outline/hierarchy.py`) ‚Äî distinct from initial full-hierarchy *creation*, which does require reasoning and stays in Agent 1

**State passing is by database reference, not shared prompt context.** Agent 1 persists its output (resolved role, grounded outline rows, confidence tiers) to Postgres. Agent 2 receives only `user_id` / `topic_id` references and reads what it needs directly from the database. Raw agent output is never passed as accumulated text into the other agent's context window ‚Äî this follows the course's explicit "Decouple State... pass only URIs or pointers" guidance and keeps each agent's context scoped and small.

**Gates are structural, not advisory ("Reviewer & Gate" node pattern).** The database write functions for outline items, patch-notes, and grounding results accept only a pre-validated object type (post-`output_guard`), not a raw agent-generated dict. An agent cannot persist an ungrounded item because the write path itself rejects anything that hasn't passed the gate ‚Äî this is a software constraint, not a prompt instruction the agent could forget.

## 3. Component Boundaries (Agent Assignment)

Two real agents, consistent with the "don't inflate agent count for rubric-checkbox reasons" principle established during requirements:

### Agent 1 ‚Äî Research & Outline Agent
**Owns (reasoning/generation only):** clarify-gate conversation (asking narrowing questions, proposing/explaining role interpretations ‚Äî the *content* of what to ask, not the round-counting), cross-validation normalization judgment (anchored to `roles_cache`), initial full-outline hierarchy creation (sequencing sourced skills into dependency order).
**Calls as tools (deterministic, not owned):** `security/input_gate.py` (bound state, first-pass real/vague/nonsense stated-goal classification, reject detection), `security/output_guard.py` (confidence/source validation before any write), `data/roles_cache.py` (I/O), `data/himalayas_parser.py` (search_jobs text-blob parsing), `data/himalayas_relevance.py` (relevance heuristic, since Himalayas cannot independently signal zero results), `data/cross_validation.py` (PRD ¬ß7.3 tier-decision rules), `data/grounding_fallback.py` (cached-fallback and general-knowledge-only floor rungs, invoked only once live Himalayas/Tavily grounding has already failed for the current request), `outline/significant_event.py` (diff logic), `outline/hierarchy.py` (insertion into existing structure), `patches/patch_manager.py` (confidence branching).
**Tools:** Himalayas MCP, Tavily search, Gemini (via `google-genai` ‚Äî clarify-gate conversational content and initial outline-hierarchy sequencing), Postgres (via the gated write paths above only ‚Äî never a raw insert).

**Resolved (`src/security/input_gate.py`'s `classify_stated_goal`, `src/agents/research_outline_agent.py`'s `begin_clarify_gate`/`advance_clarify_gate`; judgment calls made and flagged during implementation, not specified above at the time of writing):**
- **First-pass classification mechanism (`classify_stated_goal`):** purely lexical/vocabulary-based, never a market-existence check. A multi-word phrase ending in a recognized occupation noun (`_ROLE_NOUN_SUFFIXES` ‚Äî Engineer, Analyst, Manager, etc.) classifies as `REAL` unless it also contains a small, curated fabricated-title marker (`_FABRICATED_TITLE_MARKERS` ‚Äî vibes, dragon, whisperer, wizard, etc.), which is what lets a title the module has never seen before (e.g. "Site Reliability Engineer") through as real rather than gate-rejected, per PRD ¬ß7.2's niche-title instruction. A single word classifies as `VAGUE` only if it's in a coarse tech/career vocabulary (`_VAGUE_TECH_SINGLE_WORDS`); otherwise a real single word with no tech connection (PRD ¬ß7.2's "banana", "purple") is `NONSENSE`. Keyboard-mash detection combines a vowel-presence check with a max-consecutive-consonant-run threshold (`_MAX_PLAUSIBLE_CONSONANT_RUN = 4`), since a bare vowel-presence check alone cannot catch a mash string that happens to contain one vowel (e.g. "asdkjfh"). All vocabulary lists are coarse and non-exhaustive ‚Äî same status as `data/tavily_parser.py`'s `TECH_SKILL_VOCABULARY`, flagged for tuning as real usage is observed. The fabricated-title denylist in particular fails open to `REAL` for an unrecognized fabricated title, which is the deliberately safer failure direction per PRD ¬ß7.2's "never gate-reject a niche real title" instruction.
- **Interface redesign, not implemented as originally stubbed:** the pre-existing `generate_clarify_gate_response(conversation_so_far, current_round)` scaffold (a bare round-number int) predated `security/input_gate.py`'s actual `ClarifyGateStage` state machine and couldn't represent it ‚Äî a round number alone can't distinguish NARROWING from PROPOSE_BEST_GUESS from EXPLAIN_ROLE. Replaced with `begin_clarify_gate` (the first turn) and `advance_clarify_gate` (every subsequent turn, dispatching on `ClarifyGateState.stage`), threading a new `ClarifyGateContext` (the original stated goal, captured once and never overwritten, plus the most recently proposed role) alongside `security/input_gate.py`'s state.
- **Two fixed, non-LLM messages:** the nonsense re-prompt and the zero-market-signal exit message are both plain constants, not Gemini-generated ‚Äî both outcomes are already fully decided by deterministic logic (`classify_stated_goal`, `ground_role`'s confidence tier) before the message is chosen, so an LLM paraphrase would add cost/latency without adding information.
- **Model choice:** `gemini-2.5-flash` for all clarify-gate conversational turns ‚Äî short, low-latency generation, not grounded content. ~~a stronger tier may be warranted for outline/hierarchy sequencing later, a separate decision~~ ‚Äî **Superseded** (Gemini free-tier quota task, see ¬ß3's outline-hierarchy-sequencing bullet below): that stronger tier was tried (`gemini-2.5-pro`) and reverted ‚Äî this project's API key has zero free-tier quota for it. Every Gemini call in this codebase uses `gemini-2.5-flash` today.
- **`PROMPT_REGISTRY`:** five versioned prompts (narrowing question, narrowing-answer evaluation, best-guess proposal, role explanation, acceptance evaluation) ‚Äî the first real use of CLAUDE.md's LLM Call Discipline registry in this codebase; `security/input_gate.py`'s classification is deliberately *not* prompt-based at all (no registry entry needed), per its own no-LLM mandate.
- **Known gap, carried forward, not resolved by this task:** `utils/logger.py` still doesn't exist (Architecture's own prior-flagged gap for `ground_role`'s Himalayas/Tavily calls) ‚Äî these new Gemini calls likewise produce zero cost/usage log entries for now. Worth flagging loudly before any real/demo run, same as before.

**Resolved (`src/agents/research_outline_agent.py`'s `_generate_content_with_retry`; the Gemini-retry/rate-limiting task ‚Äî a real bug fix plus new infrastructure, not specified above at the time of writing):**
- **Confirmed, not assumed: every Gemini call in this codebase already routed through exactly one pair of shared functions (`_call_gemini_text`/`_call_gemini_json`, both here) before this task.** Grepped directly to confirm rather than inferred: `agents/coaching_pace_agent.py` and `.agent/skills/verification_question_generator/generator.py` both import `_call_gemini_json` directly rather than calling `client.aio.models.generate_content` themselves. The one real duplication found was *within* this file ‚Äî `_call_gemini_text` and `_call_gemini_json` each independently called `client.aio.models.generate_content` inside their own `try`/`except`. Closed by extracting a new, genuinely single low-level function, `_generate_content_with_retry`, that both now call ‚Äî retry, backoff, pacing, and error-formatting live in exactly one place, not duplicated between the two.
- **Two real, pre-existing bugs, not by-design gaps:** (1) no retry/backoff existed anywhere a Gemini call could hit a transient 429/503 ‚Äî a single transient failure surfaced immediately as a hard error to the user; (2) some exceptions (a bare `asyncio.TimeoutError`, raised directly by `asyncio.wait_for` on a real timeout) stringify to an **empty string**, which the prior `f"Gemini call failed: {exc}"` formatting turned into exactly "Gemini call failed: " with nothing after the colon. Reproduced in a dedicated test (`test_gemini_call_error_message_is_never_swallowed`), not just asserted.
- **Retry only a transient 429 (RESOURCE_EXHAUSTED)/503 (UNAVAILABLE), identified via the google-genai SDK's own `.code` attribute** (`google.genai.errors.APIError`'s `ClientError`/`ServerError` subclasses carry the real HTTP status as `.code`) ‚Äî never a 400/401/403 or any other non-transient failure, which would just waste a call on an outcome retrying can't change. Parameters: `GEMINI_RETRY_MAX_ATTEMPTS = 4` (retries beyond the first attempt), `GEMINI_RETRY_BASE_DELAY_SECONDS = 1.0`, exponential (`2**attempt`) with equal-jitter (half the computed delay, plus a random amount up to the other half), capped per-attempt at `GEMINI_RETRY_MAX_DELAY_SECONDS = 30.0` ‚Äî nominal (jitter-free) delays across 4 retries starting at 1s are 1/2/4/8s, well under this cap and under the ~30-60s total ceiling this was scoped against. Confirmed live against a real, persistent 503 (the free-tier quota already exhausted from an earlier task's testing): the full retry sequence took ~16.5s of real wall-clock time before surfacing the final, fully-diagnosable error message.
- **`_format_gemini_error`:** prefers the SDK's own structured `code`/`status`/`message` fields when present, falls back to `str(exc)` if non-empty, and to `repr(exc)` as a last resort ‚Äî the fallback chain that actually fixes bug (2) above, since `repr()` never returns an empty string the way `str()` can for some exception types.
- **Pacing guard, not a queue:** `GEMINI_MIN_CALL_INTERVAL_SECONDS = 1.0`, enforced via module-level state (`_last_gemini_call_started_at`) so a pipeline run that fires several Gemini-backed steps back to back (Clarify Gate -> Grounding -> Outline Creation) doesn't burst-call the API ‚Äî deliberately module-level, not per-caller, since Gemini's free-tier rate limit is enforced per-project across every call this process makes, not per API key (confirmed via Google's own docs). No key-rotation logic was added or considered, per this task's explicit scope.
- **`tests/conftest.py` (new file):** an autouse fixture disables the pacing guard for the whole test suite ‚Äî its state is module-level and would otherwise persist and add real wall-clock delay across unrelated tests sharing one pytest process, not just Gemini-specific ones.

**Resolved (`src/agents/research_outline_agent.py`'s `_generate_content_with_retry`/`_compute_gemini_retry_loop_timeout`; a same-session incident-driven follow-up (outer retry-loop timeout + a pacing-guard race fix), whose spec reconciliation was explicitly skipped at the time under time pressure, plus a real production bug found and fixed in a later refactor/cleanup pass ‚Äî both closed here together):**
- **An outer ceiling on the whole retry sequence, and a lock around the pacing guard, closed a real incident:** a live Outline Creation call once raised a bare `TimeoutError()` during a burst of transient 503s, because only each *individual* attempt inside the retry loop had its own timeout ‚Äî nothing bounded the whole attempt-plus-backoff sequence. `asyncio.wait_for(_run_attempts(), timeout=...)` now wraps the entire sequence. Separately, `_pace_gemini_call`'s check-then-sleep-then-update sequence is a genuine TOCTOU race even though `asyncio` is single-threaded (the `await` yields the event loop between the check and the update) ‚Äî fixed with a module-level `asyncio.Lock()` around the whole sequence, not just the read.
- **Real bug found and fixed in a later refactor/cleanup pass, reproduced live: the exact same "Outline Creation failed: Gemini call failed: TimeoutError()" incident recurred, for a different reason.** `EXTERNAL_CALL_TIMEOUT_SECONDS` (10s, sized for a short conversational turn) was also being used as the per-attempt timeout for `create_initial_outline`'s call ‚Äî a one-shot structured generation over potentially dozens of skills. A live probe (10 grounded skills -> 21 sequenced topics, `gemini-2.5-flash`) measured 13.5s end to end, already past the 10s budget; since a bare `asyncio.TimeoutError` has no `.code` attribute, `_is_retryable_gemini_error` never retries it, so the call failed on its very first attempt, deterministically, every time. Fixed with a new `HEAVY_GENERATION_TIMEOUT_SECONDS = 45` per-attempt budget, passed explicitly by every call site that generates a full multi-field/multi-item structured payload in one shot (`create_initial_outline`, `_generate_topic_explanations`, both `generate_day_content` variants, `generate_closing_note`) ‚Äî short conversational turns (clarify-gate messages, outline-confirmation turn classification) keep the original `EXTERNAL_CALL_TIMEOUT_SECONDS` default.
- **The outer retry-loop ceiling is now computed per call from the per-attempt timeout, not a single fixed constant** (`_compute_gemini_retry_loop_timeout`): a fixed 120s ceiling was sized only for 10s short-turn attempts, and would itself cut off a still-healthy heavy-generation retry sequence under a 429/503 burst (5 attempts x 45s already exceeds 120s) ‚Äî recreating the exact class of incident the outer ceiling exists to prevent. The formula reproduces the original fixed value exactly at the 10s short-turn timeout (`_compute_gemini_retry_loop_timeout(10) == 120.0`), so short-turn behavior is unchanged.
- **`timeout` defaults to `None`, not `EXTERNAL_CALL_TIMEOUT_SECONDS`, on `_generate_content_with_retry`/`_call_gemini_text`/`_call_gemini_json`'s signatures, resolved to the live module constant inside the function body.** A default bound directly in the signature is evaluated once at function-definition (import) time ‚Äî a test's `monkeypatch.setattr(roa, "EXTERNAL_CALL_TIMEOUT_SECONDS", ...)` would then silently have no effect on any caller relying on the default, a real regression caught by the existing timeout-path test suite while making this exact fix.

**Resolved (`src/agents/research_outline_agent.py`'s `reset_gemini_client_for_new_event_loop`, `src/main.py`'s `_run_async`; a real, live-reproduced "Outline Confirmation failed: Gemini call failed: Event loop is closed" incident):**
- **Real bug, reproduced in isolation before fixing:** two back-to-back `asyncio.run(_call_gemini_text(...))` calls sharing the module-level memoized `_gemini_client` (CLAUDE.md's "one client per module" convention) ‚Äî the first succeeds, the second raises `RuntimeError: Event loop is closed` deep inside httpcore's connection teardown. Root cause: `genai.Client` lazily creates and caches an async httpx transport bound to whichever event loop is running the first time `client.aio` is used; `main.py`'s `_run_async` runs every Gemini-backed call via its own fresh `asyncio.run(...)` per Streamlit rerun (correct per that function's own docstring ‚Äî each rerun is a fresh top-to-bottom script run), so the cached transport is bound to a loop that's already closed by the time the *next* rerun's call reaches it. Any two Gemini calls in the same session after the first one hit this, not just the Outline Creation -> Outline Confirmation transition specifically.
- **Fix: `reset_gemini_client_for_new_event_loop()` discards the memoized `_gemini_client`, called from `main.py`'s `_run_async` in a `finally` block after every `asyncio.run(...)` completes (success or failure).** The next `_get_gemini_client()` call then constructs a genuinely fresh client (and transport) inside its own new loop. `genai.Client(api_key=...)` is cheap to construct (holds only the API key until first real use), so paying that cost once per Streamlit rerun is not a regression of the "one client per module" convention ‚Äî it's that convention's correct grain in a fresh-event-loop-per-call architecture, not a per-request client rebuilt mid-loop.
- **Verified live, not just by unit test:** the same two-calls-sharing-one-client repro that reproduced the crash was re-run with the reset wired in (three consecutive calls, all succeeding) before writing the regression tests (`tests/test_research_outline_agent.py::test_reset_gemini_client_for_new_event_loop_forces_fresh_client`, `tests/test_main.py::test_run_async_resets_gemini_client_after_success`/`test_run_async_resets_gemini_client_even_on_failure`).

**Resolved (`src/agents/research_outline_agent.py`'s `_call_gemini_json`/`_parse_gemini_json_object`; a real, user-reported "Gemini response was not valid JSON: ..." incident on both Outline Creation and Outline Confirmation):**
- **Real bug, exact broken payload captured from a live run, not a hypothetical:** `_generate_topic_explanations`'s one-shot call over ~70 topics came back as a genuine 200 OK response with syntactically broken JSON ‚Äî a single missing `}` immediately after the first `topic_explanations` list entry, otherwise well-formed. Not a 429/503, so `_generate_content_with_retry`'s transport-level retry (`GEMINI_RETRY_MAX_ATTEMPTS`) never engaged ‚Äî `_call_gemini_json` had no retry of its own for a response that comes back but fails to parse/validate, so the entire call failed outright on this single bad generation roll. The user confirmed the same failure class had already hit Outline Creation's own large one-shot call (`create_initial_outline`'s hierarchy-sequencing JSON) in an earlier run.
- **Fix: `_call_gemini_json` now retries up to `GEMINI_JSON_RETRY_MAX_ATTEMPTS = 2` additional times when the response parses/validates via the new `_parse_gemini_json_object` helper and fails** (empty response, invalid JSON syntax, non-object JSON, or a missing required key) ‚Äî separate from, and in addition to, the existing transport-level 429/503 retry every attempt still goes through. Per CLAUDE.md's "error-fed retry" LLM Call Discipline, each retry's prompt is the original prompt plus the concrete parse/validation error from the previous attempt, not a generic "try again" ‚Äî `original_prompt` is captured once before the loop and never overwritten, so a later retry can't contaminate itself with a prior attempt's correction as if it were the original input.
- **Verified against the exact reported failure, not just a synthetic case:** re-ran the real broken JSON string from the user's error report through `_call_gemini_json` (via a hand-built fake client standing in for the real `genai.Client`, since this is a probabilistic generation defect that can't be forced from the live API on demand) followed by a well-formed response, and confirmed it now recovers on the second attempt with the parse error present in that attempt's prompt. New tests: `tests/test_research_outline_agent.py::test_gemini_json_helper_retries_on_malformed_json_and_recovers` (recovery path) plus updated exhaustion tests (`test_gemini_json_helper_raises_gemini_call_error_on_malformed_json`/`_on_missing_key`, `test_create_initial_outline_raises_on_missing_groups_key`) that now queue `GEMINI_JSON_RETRY_MAX_ATTEMPTS + 1` bad responses to still prove the loop exhausts and raises when every attempt fails.
- **Deliberately scoped to `_call_gemini_json` only, not `_call_gemini_text`.** Both reported incidents were JSON-shaped one-shot generations; `_call_gemini_text`'s only failure mode (an empty response) wasn't implicated and extending this there wasn't part of what broke.

**Resolved (`src/agents/research_outline_agent.py`'s `create_initial_outline`; judgment calls made and flagged during implementation, not specified above at the time of writing):**
- **Structural sourcing safety, not merely a prompt instruction:** Gemini's JSON response never contains `source_url`/`source_type`/`confidence` fields at all ‚Äî it only names which grounded skill (`source_skill`, by exact name) each topic came from. `create_initial_outline` re-attaches the real values by looking that name up in a map built directly from the input `core_skills`/`emerging_skills`, and raises `GeminiCallError` if `source_skill` doesn't match any input skill name. This makes "the LLM invents/drops/alters a source_url or confidence" structurally impossible rather than merely discouraged by the prompt ‚Äî the same "gates are structural, not advisory" principle ¬ß2 already states, applied to this specific risk.
- **Every grounded skill must be covered by at least one topic, or the call raises:** extending CLAUDE.md guardrail #1's "never drop" reading to mean a skill that was successfully grounded must not silently vanish from the outline either. ### Cron job (not an agent)
**Owns:** the deterministic scheduled trigger (minimum every 30 days) that invokes Agent 1's Research pipeline against the seed role list to refresh `roles.json`. Deliberately not agentic ‚Äî the trigger is wall-clock time, not judgment.

**Implementation ‚Äî two layers, one shared function:**
- **Refresh function** (single, reusable): re-runs Agent 1's Research/Grounding pipeline for the seed role list, writes results to `roles_cache`. Called by both triggers below ‚Äî no duplicated logic.
- **Primary trigger ‚Äî GitHub Actions scheduled workflow** (`schedule:` cron, e.g. every 30 days): calls the refresh function directly (either via a script with its own DB/API credentials, or by hitting an app endpoint). Genuinely wall-clock triggered, independent of app traffic ‚Äî this is the real "deployability"/"scheduled job" demonstration for the video.
- **Resilience layer ‚Äî startup/session staleness check**: on Streamlit app startup (or first session of the day), check `roles_cache.last_updated` per cached role; if past the 30-day floor, call the same refresh function inline before continuing. Ensures the system stays honest even if the GitHub Action hasn't fired yet relative to a live demo session ‚Äî not a replacement for the scheduled trigger, a safety net alongside it. **Implementation note:** the staleness comparison itself (`is_stale`) is a pure function living in `data/roles_cache.py` rather than a separate pure module ‚Äî it operates on a value only that module reads (`last_updated`) with no other caller, so a separate module would be pure indirection; it still takes an explicit `reference_time` rather than calling the wall clock internally, keeping it deterministic and testable like the project's other pure modules.

- **The "new" snapshot always treats every skill in `grounding_result.skills` as `SkillBucket.CORE_SKILLS`**, deliberately mirroring exactly what `refresh_roles_cache` just persisted via `upsert_role` (`core_skills=grounding_result.skills, emerging_skills=[]` ‚Äî the same degenerate-split workaround already flagged in PRD Future Improvements #5). Diffing against anything other than what was actually written would make "significant" disagree with what's actually sitting in `roles_cache`.
- **New data-access function, `data/outline_topics.py`'s `get_completed_topics_matching_skill(session, skill_name)`**: finds every `completed`/`completed_test_out` row, across all users, whose `topic_name` matches `skill_name` case-insensitively. Placed in the existing `data/outline_topics.py` (not a new module) since it's a plain read query over that same table, consistent with that module's existing `get_topic`/`get_topics_in_group`. Matching uses SQL-side `func.lower()` paired with Python-side `.lower()` ‚Äî not this codebase's usual `.casefold()` convention ‚Äî because the comparison must execute inside the DB query and Postgres's `lower()` only approximates ASCII lowercasing; pairing it with Python's more aggressive Unicode `.casefold()` on the other side risks a silent mismatch. Flagged as a narrow, deliberate departure, not an oversight.
- **New module, `data/patch_notes.py`** (`create_patch_note`/`get_pending_patch_notes`/`get_deferred_patch_notes`/`update_patch_note_status`), mirroring `data/roles_cache.py`'s dependency-injected-`Session` convention. `create_patch_note` requires a `ValidatedGroundedContent` (not a raw dict) for the patch's source/confidence ‚Äî CLAUDE.md guardrail #12's structural gate, applied here exactly as `data/roles_cache.py`'s `_to_skill_entry` already applies it to `roles_cache` skill entries. This module imports `PatchStatus` from `patches/patch_manager.py` (a new, previously-unestablished `data/` -> `patches/` dependency direction) purely for the status-string enum ‚Äî no functions are imported back, so this does not invert `patches/patch_manager.py`'s existing "pure, no DB I/O" role.
- **`models/schemas.py`'s `PatchNote` is now a fully mapped `Base` subclass** (previously a bare, unmapped placeholder), matching Architecture ¬ß5's `patch_notes` schema exactly. `user_id`/`origin_topic_id` remain plain `UUID` columns with no `ForeignKey` constraint ‚Äî consistent with every other cross-reference column in this codebase, since `User` is still unmapped (a FK to an unmapped table isn't possible, and partially enforcing one FK but not the other would be inconsistent).

**Known limitation (see PRD ¬ß7.9 for the user-facing framing):** the patch-note candidates `create_patch_notes_for_significant_events` writes carry a deterministic, mechanically-assembled `new_content` sentence (role name, skill name, new confidence tier only) ‚Äî not real Agent-1-authored narrative content. PRD ¬ß7.9 and this section both assign patch-note *content* authorship to Agent 1's reasoning, but the cron job is explicitly "not an agent" (this section's own heading) and generating real content here would require an LLM call this task's scope fence forbade. `_build_patch_note_content` satisfies CLAUDE.md guardrail #1 (a real `source_url`/`confidence`, both taken from the same `ValidatedGroundedContent` that crossed upward) without fabricating the narrative Agent-1 would normally write. **Not resolved by this task:** a production version routes each detected significant event through Agent 1 to generate the real explanation before the patch-note is persisted.

## 4. Agent Skill

**Verification Question Generator** ‚Äî packaged as a real Skill artifact (SKILL.md + implementation), not inline agent logic.

- **Input:** topic source material (text/URL), number of questions needed (5, or 1 for a targeted retry)
- **Output:** structured question objects ‚Äî `{question, grading_criteria, source_url}` ‚Äî schema-validated
- **Trigger description (for SKILL.md):** "Generate source-anchored comprehension questions with grading criteria from study material. Use when a topic needs verification questions (initial 5-question set, a fresh retry question, or a test-out check). Do not use for market-data grounding or general Q&A unrelated to a specific study topic."
- **Reused by:** Agent 2, for every topic's initial verification, every retry (fresh question each time), and test-out checks.
- **Why this one:** narrowest, most repeated, cleanest single-responsibility unit in the whole pipeline ‚Äî textbook "one skill, one job."

**Resolved (`.agent/skills/verification_question_generator/generator.py`; judgment calls made and flagged during implementation, not specified above at the time of writing):**
- **Grading is this Skill's job, not the caller's ‚Äî correcting an earlier draft.** An earlier `SKILL.md` draft (written before this task) stated grading was "used by the caller, not performed by this skill." This task's instructions were explicit that strict pass/fail grading, given a question's `grading_criteria` and a user's answer, is this Skill's responsibility (`grade_answer`) ‚Äî the earlier draft is superseded, not merely extended.
- **Structural sourcing safety, not merely a prompt instruction, for `source_url`:** Gemini's question-generation response never contains a `source_url` field at all ‚Äî it only produces `question_text`/`grading_criteria` per question. `generate_questions` attaches the caller's real `source_url` to every question directly, the same structural split `agents/research_outline_agent.py`'s `create_initial_outline` and outline-confirmation functions already use, applied here to a schema with no confidence-tier concept.
- **`security/output_guard.py`'s `validate_output_object` does not apply to this Skill's output** ‚Äî that function's contract requires `source_url`/`source_type`/`confidence`, and a verification question has no meaningful "confidence tier" the way a grounded skill or outline topic does. A purpose-built `SchemaValidationError`/`_validate_question_object` enforces this Skill's actual schema instead (non-empty `question_text`/`grading_criteria`/`source_url`), still honoring guardrail #1's "never fabricate/omit a source" for the one field that does apply.
- **Freshness mechanism: exact (case-folded) match, not semantic-similarity detection.** "Genuinely distinct from any previously generated question" is enforced structurally as "never a verbatim (case-insensitive) repeat" of anything in `previous_question_texts`, or of another question in the same generation batch ‚Äî `generate_questions` raises if either happens. The prompt separately instructs Gemini to make a retry question "genuinely different, never a reworded repeat," but that instruction is a best-effort quality request, not something this Skill verifies deterministically; true semantic-similarity detection (e.g. embeddings) is out of scope for this task, a known limitation (see PRD ¬ß11 / Architecture ¬ß10).
- **`mode` (`initial | retry | test_out`) dropped from the interface**, present in the pre-existing `SKILL.md` draft's input schema but never load-bearing: `num_questions` (5 vs 1) plus whether `previous_question_texts` is non-empty already fully determine this Skill's behavior ‚Äî `test_out` uses the identical 5-question path as `initial`, distinguished only by *when* the orchestrator calls it, not by anything this Skill needs to know.
- **Flagged architectural seam, not solved by this task:** `generator.py` imports `agents.research_outline_agent._call_gemini_json` (and `GeminiCallError`) directly ‚Äî an underscore-prefixed, module-private-by-convention name, reached into across a real package boundary (`.agent/skills/` -> `src/agents/`), rather than duplicating the call/timeout/error-handling logic per this task's explicit instruction. The cleaner long-term fix is extracting that Gemini-call infrastructure into a shared `src/utils/` module both the Agent and this Skill import from as legitimate peers; not attempted here, since it would mean refactoring already-tested, already-committed Agent code as a side effect of a Skill-scoped task. Flagged for a future promotion.
- **Model choice:** `gemini-2.5-flash` (`VERIFICATION_GEMINI_MODEL`, this Skill's own constant, not imported from `agents/research_outline_agent.py`) ‚Äî question generation from one piece of source material and single-answer grading are both short, bounded tasks, closer in shape to the clarify gate's per-turn calls than to outline-hierarchy sequencing's whole-curriculum reasoning.
- **`PROMPT_REGISTRY` is this Skill's own, separate from `agents/research_outline_agent.py`'s** ‚Äî two new versioned prompts, `verification_question_generation_v1` and `verification_answer_grading_v1`. CLAUDE.md's LLM Call Discipline says "a module-level `PROMPT_REGISTRY`", not one project-wide singleton, and this Skill is architecturally independent (packaged as "a real Skill artifact... not inline agent logic").

## 8. Confidence Ladder ‚Äî canonical values

Used consistently across `roles_cache`, `outline_topics`, `patch_notes`:
`high` ‚Üí `medium` ‚Üí `low` ‚Üí `cached-low` ‚Üí `general-knowledge-only` ‚Üí (reject, no record created)

**Resolved (`src/data/grounding_fallback.py`; judgment calls made and flagged during implementation, not specified above at the time of writing):**
- **`general-knowledge-only` is structurally distinct from `ValidatedGroundedContent`, never coerced into it.** This rung means "no real source of any kind" by definition (PRD ¬ß7.3) ‚Äî `security/output_guard.py`'s `ValidatedGroundedContent` requires a non-empty, structurally valid `source_url`, and fabricating one to force this rung through that type would violate guardrail #1 and the "never silently fabricate a source" rule. `grounding_fallback.py` instead returns a separate `GeneralKnowledgeFloorResult` (role_name, confidence, a human-readable label) with no `source_url` field to omit or fake. Per `specs/scenarios/high_risk_flows.feature`'s "No source returns usable data" scenario, this result is for honest user-facing reporting only and must never be written into `outline_topics`/`patch_notes`/`roles_cache`.
- **A stale `roles_cache` entry (past the 30-day floor) still counts as usable cached-fallback data**, not as equivalent to "no entry." Staleness only triggers the existing cron/startup refresh cycle; it does not disqualify already-grounded data from being served as today's fallback once live sources have failed. `is_stale` is surfaced as metadata on `CachedFallbackResult` so a caller can label the result honestly (e.g. "based on data from 47 days ago"), per PRD ¬ß7.3's "labeled with `last_updated`" phrasing. Flagged as a genuine judgment call, revisable if a stricter reading (stale should escalate straight to the floor) is preferred.
- **`roles_cache` skill entries carry no per-skill `source_type` on disk** (¬ß5's JSONB shape is `{skill, source_url, confidence}` only), so one cannot be read back honestly. Cached-fallback results are stamped with a constant `source_type` of `"roles_cache-cached"`, naming the provenance layer truthfully (a previously-validated result now being re-served from cache) rather than guessing at the original external source's type.

**Known limitation (see PRD ¬ß7.2/¬ß7.3 for the full finding):** the `reject` rung's "zero signal" trigger cannot rely on Himalayas independently reporting zero results. Live testing during `src/data/himalayas_parser.py`'s development (nonsense keyword, extreme `salary_min`, obscure `country` + `exclude_worldwide` combination) never produced a genuine empty `search_jobs` response ‚Äî it fell back to broad/unrelated matches instead. Cross-validation's zero-signal determination therefore rests on Tavily + `roles_cache`, with any Himalayas results needing a relevance judgment rather than an empty-check. **Resolution mechanism below (`src/data/himalayas_relevance.py`, `src/data/cross_validation.py`); the underlying limitation itself still stands, only the inference mechanism around it is new.**

**Resolved (`src/data/himalayas_relevance.py`, `src/data/cross_validation.py`, `src/agents/research_outline_agent.py`'s `ground_role`; judgment calls made and flagged during implementation, not specified above at the time of writing):**
- **Relevance heuristic (`src/data/himalayas_relevance.py`):** infers "no usable Himalayas signal" from a non-empty `search_jobs` response by checking title-token overlap against the searched role. `compute_title_relevance` scores each listing (fraction of the role's tokens found in the title, simple lowercase word-set overlap, no stemming/synonyms); a listing counts as relevant at or above `PER_LISTING_TOKEN_OVERLAP_THRESHOLD` (0.5). The fraction of relevant listings required to trust the whole batch is banded by result count ‚Äî `MIN_COUNT_THRESHOLD` (5) and below require `MIN_COUNT_RELEVANCE_FRACTION` (0.6); `MAX_COUNT_THRESHOLD` (25) and above require only `MAX_COUNT_RELEVANCE_FRACTION` (0.2); linearly scaled in between ‚Äî mirroring `pace/calculator.py`'s `TIMING_OUTLIER_THRESHOLD`/`TIMING_SATURATION_DEVIATION` banding pattern. All five constants are unvalidated calibration judgment calls, flagged for tuning once real cross-validation runs are observed, exactly like the pace calculator's constants.
- **Cross-validation tier decision (`src/data/cross_validation.py`):** a pure function applying PRD ¬ß7.3's rules ‚Äî both sources agree with the `roles_cache` anchor ‚Üí `high`; single source (Himalayas or, now, Tavily ‚Äî see below) ‚Üí `medium`; both sources have signal but Himalayas's skills don't overlap the anchor ‚Üí `medium` with an explicit `has_conflict=True` flag (a genuine conflict, PRD's "flagged" outcome); no anchor at all (Himalayas has signal) ‚Üí `low` (PRD's niche/no-anchor rule ‚Äî the LLM sanity-check pass that rule also describes remains explicitly deferred, not implemented); no usable signal from *either* source ‚Üí `reject`. Anchor "agreement" requires at least `ANCHOR_OVERLAP_MINIMUM` (1) shared skill between Himalayas's extracted skills and the roles_cache anchor.
- **Tavily-only signal, resolved:** what was previously a hard scope limit (`himalayas_has_signal` as an unconditional precondition for any tier above `reject`) is now resolved. A Tavily-only batch (Himalayas has no usable signal) reaches `medium` confidence if it clears `TAVILY_DISTINCT_SKILLS_TRUST_THRESHOLD` (3 *distinct* skills ‚Äî not total mentions, not count of skill-bearing results ‚Äî found across the whole batch via `src/data/tavily_parser.py`'s extraction; unvalidated calibration judgment call, same status as the other constants on this page). Tavily's own `score` field is used **only** to select which already-skill-bearing result becomes the citation `source_url` (the highest-scoring one among those with ‚â•1 extracted skill) ‚Äî never to decide trust, per `data/tavily_parser.py`'s real-data finding that `score` does not predict extractability. All distinct skills found across the whole trust-qualifying batch are written, attributed to that one citation URL (`TavilyCitation`) ‚Äî not just the citation result's own skills; see `data/cross_validation.py`'s `TavilyCitation` docstring for why per-result attribution isn't attempted.
- **Known limitation, explicitly named (not resolved by this task):** `data/tavily_parser.py`'s `TECH_SKILL_VOCABULARY` was derived from skills `data/himalayas_parser.py` already extracted for the same 4 seed roles (Backend Engineer, Frontend Engineer, Data Analyst, DevOps Engineer), plus 2 manually-added terms ‚Äî the Tavily trust check above can therefore only recognize skills Himalayas already surfaced for a role, not independently discover new ones. For any role outside those 4 (including the rest of PRD ¬ß7.3's seed list, e.g. AI/ML Engineer), Tavily's trust check has no special vocabulary coverage and will likely under-count real signal. **Future improvement path:** a real, independently-sourced skill vocabulary (not derived from Himalayas), or a smarter extraction method (e.g. an LLM-assisted pass, consistent with PRD ¬ß7.0's "reserve LLM judgment for genuinely ambiguous cases") ‚Äî not attempted here.
- **Explicit attribution model for Tavily-sourced skills (previously only implicit in the bullet above):** a Tavily-sourced skill's `source_url` represents the trust-qualifying *batch* it was found in, not a per-skill provenance guarantee. Concretely: an individual skill's cited `source_url` may be a result that did *not itself* mention that specific skill ‚Äî it mentions some skill(s) in the batch and won the citation ranking (highest `score` among skill-bearing results), while the skill actually being attributed to it may have been extracted from a *different* result in the same batch. The URL is always real and the skill was always genuinely found somewhere in that same trust-qualifying batch, for that role search ‚Äî this is not fabrication ‚Äî but it is a meaningfully different attribution model from Himalayas's, and is a new pattern in this codebase: `ground_role`'s Himalayas path (`himalayas_skill_map`, `src/agents/research_outline_agent.py`) attributes each skill to the specific listing it was actually extracted from ‚Äî genuine per-skill provenance ‚Äî with no precedent for the batch-level citation Tavily uses.
- **Fallback-only-on-failure ordering, now enforced:** `ground_role` (`src/agents/research_outline_agent.py`) only calls `data/grounding_fallback.py`'s `get_cached_fallback`/`get_general_knowledge_floor` when `data/cross_validation.py` resolves to `reject` ‚Äî previously this ordering was documented as an assumption (see the earlier `grounding_fallback.py` task) but nothing in the codebase actually called live sources first; this is the first caller that does.
- **New exception, `utils.exceptions.GroundingSourceCallError`:** raised for a genuine Himalayas/Tavily call failure (connection error, non-success response, malformed response body, or exceeding `EXTERNAL_CALL_TIMEOUT_SECONDS` = 10s) ‚Äî deliberately distinct from that source legitimately returning no relevant results, which is not an error. Both are collapsed to "no signal from this source" for tier-decision purposes, but `ground_role`'s internal `_safe_fetch_*` helpers track which happened (`"signal"` / `"no_signal"` / `"call_failed"`) so the distinction is intentional, not lost.
- **Skills written carry a source-specific `source_type`:** every `ValidatedGroundedContent` `ground_role` produces carries the confidence tier `data/cross_validation.py` decided, and either `source_type="job_listing"` (Himalayas-sourced, deduplicated by casefolded skill name across all relevant listings) or `source_type="web_search"` (Tavily-sourced, the new citation-based path above) ‚Äî never both in the same result, since the two paths are mutually exclusive (Tavily-sourced skills only appear when Himalayas had no signal at all).

## 9. Significant Event Detection ‚Äî implementation note

Deterministic diff on `roles_cache` between refreshes: for each skill, compare bucket membership (`core_skills` / `emerging_skills` / absent) and confidence tier between the old and new snapshot. Any *upward* crossing generates a patch-note candidate for every user with a completed topic matching that skill. Downward crossings are diffed but discarded (no action) ‚Äî never deleted from history, just not acted upon. **Wired into `src/cron/refresh_roles.py` ‚Äî see ¬ß3's Cron job "Resolved" block for the mechanism.**

**Resolved (`patches/patch_manager.py`'s `decide_patch_delivery`/`PatchDeliveryDecision`/`PatchDecisionState`/`resolve_patch_decision`; the delivery/surfacing-decision task ‚Äî judgment calls made and flagged during implementation, not specified above at the time of writing):**
- **`decide_patch_delivery(pending_patches, current_hierarchy_position)` decides at most one action per call ‚Äî never both branches at once.** Every pending patch is branched via the existing `branch_by_confidence` (not reimplemented). If one or more prioritize, the single one surfaced is the earliest in hierarchy/dependency order among the prioritized set; if none prioritize but at least one needs a user decision, the earliest of those is surfaced for an "ask" instead. Anything not chosen this call stays `PENDING`, reconsidered on a future call.
- **Ordering reuses `patches/patch_manager.py`'s own pre-existing `order_pending_items`, not a mechanism from `outline/hierarchy.py`.** The task's instruction was to reuse whatever ordering mechanism `outline/hierarchy.py` exposes for this rather than build a second scheme ‚Äî but `outline/hierarchy.py` itself exposes no generic "order these items" function (only `insert_new_topic`/`augment_existing_topic`, insertion/augmentation operations). `order_pending_items` (already present in `patch_manager.py`, already tested, sorting purely by `hierarchy_position`) already *is* the reusable mechanism sharing that same ordering concept ‚Äî reused directly rather than reaching into `hierarchy.py` for something that isn't there.
- **`insert_at_position` on an `"insert_now"` decision is exactly the caller's own `current_hierarchy_position`, echoed back** ‚Äî not a position this function derives independently. A patch-note's origin topic is already completed and behind the user; "insert at/near the user's current position" means the user's *current* position, not the origin topic's old one.
- **`PatchDecisionState` has no bounded-round counter, unlike `security/input_gate.py`'s `ClarifyGateState`/`OutlineConfirmationState` it otherwise mirrors** ‚Äî PRD ¬ß7.9 describes one learn-now-or-defer decision per patch-note, not a multi-round loop, so a `resolved: bool` flag is sufficient. There is deliberately no `begin_*` constructor: the dataclass's own field defaults already are the initial, unresolved state.
- **`resolve_patch_decision` is pure ‚Äî it does not write to the DB.** It only computes the new `status`/`resolved_at`; the caller applies that via the new `data/patch_notes.py`'s `update_patch_note_status`. Generates no conversational prompt text and makes no LLM call for the "learn now or defer" ask itself ‚Äî explicitly out of scope for this task, no UI/orchestration layer exists yet.
- **Not wired into any Streamlit app or `main.py`**, per this task's explicit scope fence ‚Äî `decide_patch_delivery`/`resolve_patch_decision` are plain, independently callable/testable functions only.

- Never call an external API (Gemini, Tavily, Himalayas, Neon) without an explicit timeout

## 13. Behavior Scenarios (Gherkin, high-ambiguity-risk flows only)

Per course guidance that BDD/Gherkin scenarios remove the ambiguity that causes agents to guess ‚Äî applied selectively to the flows where a wrong guess is costliest, not exhaustively to the whole system:

```gherkin
Feature: Clarify Gate bounded resolution

  Scenario: Vague-but-genuine input resolves within the round bound
    Given a user enters a vague goal like "I want to make apps"
    When the gate asks a narrowing question
    And the user answers with more specificity
    Then the gate either accepts a resolved role or asks one more narrowing question
    And the total narrowing rounds never exceed 2

  Scenario: User rejects the proposed interpretation twice
    Given the bounded rounds are exhausted
    And the system has proposed a best-guess role and the user rejected it
    And the system explained the role clearly and the user rejected it again
    When the grounding check runs on the user's own words
    Then if any market signal is found, the system proceeds at low confidence
    And if zero market signal is found, the system exits and builds no outline

Feature: Confidence Ladder enforcement

  Scenario: Both sources agree
    Given Himalayas and Tavily both return the same skill for a role
    When the Research Agent cross-validates the result
    Then the outline item is tagged confidence "high"

  Scenario: No source returns usable data
    Given Himalayas and Tavily both fail or return nothing for a role
    And roles_cache has no entry for the role
    When the Research Agent attempts to ground the outline
    Then no outline item is created
    And the system reports the general-knowledge-only floor explicitly to the user

Feature: Outline content is never removed

  Scenario: User sustains a "behind" pace
    Given a user's rolling-window pace is sustained below the drift threshold
    When the Coaching Agent triggers a pacing adjustment
    Then the outline's topic list is unchanged
    And only the day-by-day delivery schedule is extended

Feature: Verification retry cap is exactly 3 attempts

  Scenario: User fails a question twice, passes on the third attempt
    Given a user answers a verification question incorrectly on attempt 1
    And a fresh regenerated question incorrectly on attempt 2
    When the user answers a fresh regenerated question correctly on attempt 3
    Then the question is marked passed at full credit
    And exactly 3 question-generation calls have occurred for that question slot, not 4
    And the first attempt counted as attempt 1, not as a call made before the retry loop began

  Scenario: User fails all 3 attempts
    Given a user answers incorrectly on attempts 1, 2, and 3 for the same question slot
    When the retry cap is reached
    Then the system teaches the answer inline, citing the source material
    And the question is marked passed at half credit, not left unresolved
    And no fourth question-generation attempt occurs
```

**`specs/scenarios/high_risk_flows.feature` is the canonical, version-controlled copy** ‚Äî the block above mirrors it for readability in this document, but the `.feature` file is what implementation and any future scenario-runner should reference.
